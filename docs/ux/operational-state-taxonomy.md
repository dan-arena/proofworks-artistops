# Operational State Taxonomy

## Purpose

This document defines the user-facing operational state model for the ProofWorks ArtistOps platform.

The purpose of this taxonomy is to ensure:

- consistent operational state communication
- artist-friendly workflow visibility
- explainable operational escalation
- predictable dashboard behavior
- scalable state semantics
- emotionally sustainable operational guidance
- architecture-aligned operational presentation

This document defines:

- user-facing operational states
- state semantics
- state hierarchy behavior
- escalation meaning
- visual emphasis philosophy
- actionable vs informational state distinctions
- risk communication standards

This taxonomy intentionally favors:

- operational clarity
- emotional readability
- explainable escalation
- artist-friendly semantics
- contextual operational visibility

over:

- engineering lifecycle terminology
- infrastructure semantics
- technical state exposure
- emotionally hostile escalation
- enterprise workflow jargon

This document governs:

- dashboard state presentation
- operational visibility semantics
- release health communication
- warning classification
- operational escalation visibility
- workflow prioritization language

---

# Core Taxonomy Philosophy

Operational states exist to communicate:

> meaningful operational understanding

to artists.

Operational states should help users understand:
- release momentum
- operational health
- timeline confidence
- attention requirements
- workflow readiness
- release progress

Operational states should NOT expose:
- backend workflow complexity
- infrastructure orchestration details
- engineering system semantics
- internal implementation terminology

The platform should communicate:
- operational meaning
- emotional clarity
- actionable context

rather than technical mechanics.

---

# State Philosophy

Operational states should remain:

- understandable
- emotionally readable
- operationally meaningful
- explainable
- visually consistent
- artist-friendly

The platform should avoid:
- catastrophic terminology
- engineering jargon
- emotionally manipulative severity
- exaggerated urgency

Operational states should communicate:
- release condition
- operational trajectory
- attention significance
- release confidence

without creating:
- panic
- blame
- emotional exhaustion

# Top-Level Operational States

## Inactive

### Meaning

The release exists but is not currently operationally active.

Examples:
- future queued releases
- intentionally dormant projects
- releases not yet entering active planning

### Characteristics

- informational state
- low urgency
- no active operational escalation
- intentionally inactive

Inactive should NOT imply:
- failure
- neglect
- operational concern

---

## On Track

### Meaning

The release is progressing within acceptable operational confidence thresholds.

Examples:
- timeline health remains acceptable
- operational milestones remain achievable
- no significant release concerns currently exist

### Characteristics

- healthy operational state
- positive momentum
- no elevated escalation required

On Track should communicate:
- operational confidence
- healthy progression
- release stability

without excessive celebration or emotional exaggeration.

---

## Waiting

### Meaning

The release is currently waiting on:
- external dependency completion
- external review
- collaborator action
- distributor response
- third-party deliverables

### Characteristics

- informational state
- not inherently negative
- not automatically artist failure
- may still contain timeline sensitivity

Examples:
- awaiting album artwork
- distributor review pending
- collaborator stem delivery pending

Waiting should NOT imply:
- blame
- operational failure
- artist negligence

---

## Action Needed

### Meaning

The release currently requires meaningful artist attention or operational action.

Examples:
- metadata review required
- artwork approval pending
- final audio upload needed
- release scheduling incomplete

### Characteristics

- actionable operational state
- user attention appropriate
- workflow progress currently depends on action

Action Needed should communicate:
- operational relevance
- workflow dependency
- actionable next steps

without:
- panic
- emotional aggression
- catastrophic urgency

---

## At Risk

### Meaning

Release success confidence has fallen below the comfortable operational threshold.

At Risk communicates:

> declining release confidence

not:
- guaranteed failure
- artist fault
- irreversible release damage

### Characteristics

- elevated operational concern
- timeline confidence degradation
- meaningful escalation visibility
- release viability concern

At Risk may occur because of:
- internal workflow delays
- external dependency delays
- compressed timelines
- unresolved operational concerns
- collaborator bottlenecks

At Risk does NOT necessarily imply:
- immediate artist fault
- direct artist responsibility
- immediate mandatory action

However:
- artist awareness becomes important
- contingency evaluation may become appropriate
- operational attention should increase

---

## Delayed

### Meaning

The release can no longer reasonably achieve the originally intended operational timeline.

### Characteristics

- operational timeline disruption
- meaningful schedule impact
- workflow replanning likely required

Delayed should communicate:
- schedule reality
- operational consequence
- planning impact

without:
- emotional punishment
- catastrophic framing
- workflow shame

---

## Cancelled

### Meaning

The release is no longer planned to continue operationally.

### Characteristics

- inactive terminal operational state
- no active escalation
- no active workflow expectation

Cancelled should remain:
- emotionally neutral
- operationally clear
- historically understandable

The platform should avoid:
- failure-oriented framing
- emotional negativity
- punitive semantics

---

## Staged

### Meaning

The release is operationally prepared and awaiting final release execution or scheduled activation.

Examples:
- distribution completed
- approvals completed
- assets finalized
- release queued for launch

### Characteristics

- operational readiness
- high completion confidence
- pre-release holding state

Staged should communicate:
- preparedness
- release readiness
- operational completion momentum

without sounding:
- technical
- infrastructure-oriented
- deployment-focused

---

## Released

### Meaning

The release has successfully launched publicly.

### Characteristics

- completed release lifecycle state
- achievement-adjacent operational state
- analytics and milestone tracking active

Released should communicate:
- operational completion
- public availability
- release activation

without excessive hype language.

---

# Informational vs Actionable States

Operational states should distinguish between:

## Informational States

Examples:
- Inactive
- Waiting
- Staged
- Released

These states primarily communicate:
- operational understanding
- release visibility
- workflow context

Informational states should generally:
- remain visually calmer
- avoid escalation-heavy presentation
- minimize emotional intensity

---

## Actionable States

Examples:
- Action Needed
- At Risk
- Delayed

These states communicate:
- elevated operational significance
- increased attention importance
- release confidence impact

Actionable states may receive:
- stronger visual emphasis
- elevated dashboard visibility
- increased notification prominence

Visual escalation should remain:
- proportional
- explainable
- emotionally sustainable

---

# State Consistency Philosophy

Visual consistency should remain the default operational expectation.

However:

Breaking consistency intentionally may communicate:
- elevated importance
- increased operational urgency
- meaningful release concern

Visual escalation should feel:
- earned
- trustworthy
- operationally justified

The platform should avoid:
- constant urgency inflation
- meaningless visual escalation
- perpetual high-alert interfaces

Users should learn that:
> stronger emphasis actually matters

---

# Risk Philosophy

Risk states communicate:

> declining release confidence

not:
- guaranteed failure
- deterministic prediction
- irreversible outcome

The system should communicate:
- operational probability
- timeline concern
- release viability pressure

without overstating certainty.

Preferred framing:

> “This release timeline may be at risk.”

Avoid:

> “This release will fail.”

---

# External Dependency Philosophy

External dependencies may still create:
- release risk
- operational uncertainty
- timeline pressure

even when:
- the artist already completed their own work

Examples:
- waiting on graphic designer
- collaborator delay
- distributor approval delay
- external review bottlenecks

The platform should recognize:
- external dependency pressure is still operationally meaningful

while avoiding:
- unfair blame assignment
- emotionally hostile escalation

---

# Waiting vs At Risk

Waiting and At Risk should remain distinct concepts.

A release may simultaneously be:
- Waiting
and
- At Risk

Example:
- waiting on artwork delivery
- release deadline approaching
- operational confidence declining

This distinction preserves:
- operational realism
- emotional fairness
- explainable escalation

Waiting communicates:
- current dependency state

At Risk communicates:
- declining confidence trajectory

These concepts should not be conflated.

---

# State Transition Philosophy

Operational state transitions should remain:
- explainable
- predictable
- operationally meaningful
- emotionally understandable

The platform should avoid:
- sudden unexplained escalation
- opaque severity jumps
- hidden operational evaluation

Users should generally understand:
- why a state changed
- what operational condition triggered the change
- what action may improve release confidence

---

# Dashboard Prioritization Philosophy

Dashboard prioritization should consider:
- operational consequence
- release confidence
- timing pressure
- dependency significance
- actionability

The platform should prioritize:
- meaningful operational risk
- declining release confidence
- actionable operational concerns

over:
- simplistic chronological ordering
- raw task counts
- arbitrary escalation rules

---

# Emotional Sustainability Philosophy

Operational states should remain emotionally sustainable for long-term use.

The platform should avoid:
- panic-oriented escalation
- guilt-inducing terminology
- catastrophic operational framing
- emotional exhaustion loops

The platform should communicate:
- operational honesty
- realistic consequence
- actionable context
- explainable guidance

while preserving:
- artist trust
- emotional balance
- operational clarity

---

# Architecture Alignment

Operational state presentation must align with established architecture principles.

Including:
- deterministic operational evaluation
- backend-owned operational truth
- explainable operational logic
- compositional dashboard behavior

Frontend UX should:
- communicate operational meaning
- visualize operational state
- guide user understanding

The frontend should NOT:
- invent operational meaning
- create hidden state semantics
- imply unsupported certainty
- own operational evaluation rules

Operational truth belongs in backend systems.

---

# Final Taxonomy Philosophy

The ArtistOps operational state taxonomy should help artists:

- understand release condition
- understand release momentum
- identify meaningful concerns
- recognize declining release confidence
- prioritize operational attention
- maintain release awareness

while preserving:
- emotional sustainability
- artist autonomy
- operational clarity
- explainable escalation
- trustworthy guidance

The platform should feel less like infrastructure monitoring software and more like calm operational guidance from someone who understands how difficult and emotionally demanding independent music releases can be.
