# Operational Dashboard Guidelines

## Purpose

This document defines dashboard behavior, operational navigation philosophy, and dashboard interaction standards for the ProofWorks ArtistOps platform.

The purpose of these guidelines is to ensure:

- consistent dashboard behavior
- maintainable operational UX evolution
- artist-centered workflow navigation
- low cognitive overhead
- operational clarity
- explainable interaction behavior
- architecture-aligned frontend composition
- scalable dashboard governance

These guidelines intentionally favor:

- operational guidance
- calm navigation
- contextual visibility
- actionable summaries
- artist autonomy
- rapid operational scanning
- emotionally sustainable workflows

over:

- enterprise dashboard density
- workflow overload
- notification aggression
- excessive inline interactions
- operational intimidation
- dashboard chaos

This document operationalizes the principles established in:

```text
/docs/ux/experience-design-principles.md
```

---

# Core Dashboard Philosophy

The ArtistOps dashboard should function as:

> an operational navigation hub

The dashboard exists to help users:

- identify operational concerns
- understand release state
- locate meaningful workflows quickly
- identify elevated operational risk
- review release progress
- navigate intentionally between operational areas

The dashboard should NOT behave like:

- a giant task list
- a workflow prison
- a notification wall
- a project management board
- an enterprise reporting console

The home experience should remain:

- approachable
- scannable
- calm
- operationally trustworthy
- easy to navigate across devices

---

# Dashboard Intent Philosophy

Users may enter ArtistOps for many different reasons.

Examples include:

- checking release performance
- reviewing release health
- managing active releases
- reviewing metrics
- completing operational tasks
- checking schedules
- reviewing achievements
- reviewing risks
- lightweight operational check-ins

The dashboard should support:

- multiple user intents
- intentional navigation
- rapid context switching

The dashboard should NOT assume:

- every user session is operational work
- every user wants task-first workflows
- every user wants operational escalation immediately

---

# Dashboard Structure Principles

Dashboard organization should be concern-oriented.

Examples may include:

- Current Release
- Active Releases
- To Do List
- Action Calendar
- Risk & Warning Center
- Release Metrics
- Milestones
- Green Room
- Archive

Operational concerns should remain:

- visually separated
- clearly identifiable
- operationally understandable
- rapidly scannable

The dashboard should avoid:

- giant blended operational surfaces
- infinite scrolling operational feeds
- overloaded mixed-context panels

---

# Dashboard Card Philosophy

Dashboard cards should function as:

> operational summary and navigation surfaces

Dashboard cards should provide:

- concise operational visibility
- immediately meaningful status information
- lightweight contextual awareness
- clear navigation affordances

Cards should contain enough information to:

- support rapid scanning
- communicate operational state
- help users decide where to navigate

Cards should NOT attempt to:

- replace full workflow screens
- contain dense operational detail
- become miniature applications
- support complex inline workflows

---

# Dashboard Card Content Standards

Cards may include:

- release status
- progress indicators
- risk counts
- next important action
- upcoming deadlines
- milestone visibility
- lightweight metrics summaries
- operational health summaries

Cards should avoid:

- excessive descriptive text
- large operational narratives
- deep workflow interactions
- large forms
- operational clutter

The dashboard should prioritize:

> rapid situational awareness

over:

> operational execution complexity

---

# Dashboard Interaction Philosophy

The home dashboard should prioritize:

- scanning
- orientation
- navigation
- operational awareness

Operational execution should primarily occur inside:

- dedicated workflow areas
- release detail pages
- operational domain screens

This preserves:

- dashboard clarity
- mobile usability
- cognitive simplicity
- navigational trustworthiness

---

# Dashboard Ordering Philosophy

Dashboard ordering may adapt dynamically based on:

- operational risk
- release viability concerns
- dependency pressure
- timing sensitivity
- elevated workflow urgency

However:

Users should be allowed to disable dynamic dashboard ordering and preserve stable dashboard positioning if preferred.

This supports:

- spatial memory
- navigation consistency
- cognitive familiarity
- long-term usability comfort

The platform should balance:

- intelligent prioritization
- user-controlled consistency

without creating:

- unpredictable dashboard movement
- orientation fatigue
- operational confusion

---

# Risk & Warning Center Philosophy

The Risk & Warning Center should act as:

> an operational forecasting and escalation surface

The purpose of this area is NOT merely to show overdue tasks.

The purpose is to identify:

- future release viability risk
- operational bottlenecks
- dependency pressure
- timing-sensitive concerns
- likely release disruption scenarios

Risk visibility should focus on:

- operational consequence
- timeline impact
- probability of release disruption

rather than simplistic due-date ordering.

---

# Risk Visibility Standards

When operational risks exist:

- risk visibility should become visually prominent
- risk counts should be immediately visible
- severity should remain visually understandable
- affected releases should be identifiable quickly

Risk visibility should remain:

- informative
- actionable
- explainable

The system should avoid:

- alarm fatigue
- emotional hostility
- constant visual aggression
- exaggerated urgency signaling

---

# Operational Forecasting Philosophy

Warnings should communicate:

- what is at risk
- why the concern matters
- what action would reduce the risk
- what outcome may occur if ignored

Preferred example:

> Artwork approval should be completed today to maintain your planned release timeline.

Avoid:

> Task overdue.

The platform should prioritize:

> predictive operational guidance

rather than simplistic operational reminders.

---

# Risk Escalation Philosophy

Attention escalation should scale proportionally to operational risk.

The system should generally remain:

- calm
- navigable
- respectful of user intent

Escalation intensity should increase only when:

- release viability is threatened
- important operational dependencies are approaching failure
- timing pressure becomes operationally meaningful

The platform should guide urgency.

The platform should NOT enforce obedience.

---

# User Autonomy Rules

Users must retain operational ownership at all times.

The platform may:

- warn
- guide
- prioritize
- elevate visibility
- forecast consequences

The platform may NOT:

- block navigation
- lock workflows
- force completion
- require operational duplication
- trap users inside escalation flows

Users should be allowed to:

- manually complete tasks
- dismiss warnings
- acknowledge externally completed work
- manage workflows outside the platform

---

# Warning Dismissal Philosophy

Warning dismissals should behave as:

> intentional user acknowledgements

not:
- silent deletion
- hidden suppression
- permanent erasure

Dismissals should:

- require confirmation
- preserve historical traceability
- remain reviewable later
- clearly indicate the user intentionally dismissed the concern

The platform should preserve:

- operational transparency
- warning history
- causal traceability

without becoming:

- punitive
- guilt-oriented
- emotionally manipulative

---

# Historical Warning Visibility

Dismissed warnings should remain accessible through historical operational visibility areas.

The platform should support:

- operational hindsight
- historical traceability
- explainable release outcomes
- reviewable operational decision history

The system should preserve the distinction between:

- active concerns
- dismissed concerns
- resolved concerns

---

# Dashboard Tone Philosophy

The top-level dashboard should remain:

- operationally calm
- emotionally balanced
- professionally supportive
- trustworthy
- approachable

The dashboard should avoid becoming:

- emotionally noisy
- excessively celebratory
- operationally stressful
- visually chaotic

Operational trust should remain the dominant emotional tone.

---

# Milestone Recognition Philosophy

The platform should acknowledge meaningful artist achievements.

Examples include:

- stream milestones
- follower milestones
- release anniversaries
- playlist achievements
- chart milestones
- certification milestones

Milestone acknowledgements may appear as:

- dismissible overlays
- lightweight celebratory interactions
- contextual milestone callouts

These celebrations should:

- feel rewarding
- feel authentic
- remain tasteful
- avoid emotional manipulation

Milestones should acknowledge artist effort without overwhelming operational workflows.

---

# Celebration Context Rules

Operational dashboards and achievement spaces should remain emotionally distinct.

The operational dashboard should remain:

- calm
- navigable
- workflow-oriented

Metrics and achievement-focused areas may become:

- more celebratory
- more emotionally energetic
- more gamified
- more motivational

This distinction should remain intentional.

---

# Green Room Philosophy

ArtistOps should support a dedicated reflective achievement space referred to conceptually as:

> The Green Room

The Green Room exists separately from operational workflow management.

Its purpose is to allow artists to:

- reflect on accomplishments
- review career milestones
- celebrate progress
- revisit meaningful release achievements
- display recognition artifacts
- track long-term artistic growth

The Green Room should feel:

- personal
- rewarding
- reflective
- emotionally encouraging

without interfering with operational workflow management.

---

# Achievement Persistence Philosophy

Achievements should persist permanently as historical artist accomplishments.

Achievements are not:

- disposable notifications
- temporary dopamine moments
- short-lived toast messages

Achievements should function as:

- career artifacts
- milestone records
- emotional reinforcement anchors
- historical progress markers

Examples may include:

- stream milestones
- playlist achievements
- certification milestones
- release anniversaries
- follower milestones
- chart milestones

---

# Achievement Display Philosophy

Achievement systems may eventually support:

- virtual trophies
- plaques
- framed records
- milestone memorabilia
- customizable achievement displays
- historical release retrospectives

Achievement displays should emphasize:

- accomplishment
- artistic journey
- milestone context
- release history
- long-term progress

Analytics should feel connected to artist history rather than isolated numerical reporting.

---

# Theme & Accessibility Philosophy

Dashboard themes should support:

- accessibility
- cognitive comfort
- operational readability
- emotional usability

Supported customization may include:

- dark mode
- light mode
- curated notification color schemes
- curated accessibility-focused palettes
- color-blind-friendly palettes

Theme customization should remain:

- intentionally curated
- operationally readable
- semantically consistent

The MVP should avoid:

- unrestricted branding systems
- fully custom user-created themes
- uncontrolled severity coloration

Operational severity signaling must remain understandable across all supported themes.

---

# Cross-Device Dashboard Principles

Dashboard experiences should remain usable across:

- desktop
- tablet
- mobile devices

Responsive behavior should preserve:

- navigation clarity
- operational grouping
- card discoverability
- touch-friendly interaction behavior
- risk visibility
- operational hierarchy

The dashboard should not rely on:

- hover-only interactions
- desktop-only density assumptions
- hidden operational pathways

---

# Architecture Alignment

Dashboard implementation must align with established architecture principles.

Including:

- dumb UI / smart backend
- backend-owned operational truth
- frontend composition responsibility
- modular operational decomposition
- compositional dashboard behavior

The dashboard should compose and visualize operational state.

The dashboard should NOT become:

- a workflow engine
- a business rule authority
- a lifecycle calculation system
- a hidden orchestration layer

Operational truth belongs in backend systems.

---

# Final Dashboard Philosophy

The ArtistOps dashboard should help artists:

- understand what matters
- identify operational risk
- navigate release workflows quickly
- review meaningful progress
- celebrate accomplishments appropriately
- maintain release momentum

while preserving:

- calm operational cognition
- emotional sustainability
- artist autonomy
- workflow trustworthiness
- operational clarity

The dashboard should feel less like enterprise operations software and more like checking in with a trusted release manager who understands both the operational and emotional realities of independent music releases.
